# RondoSync ドキュメント改善メモ

Mintlify 向けドキュメント一式を精査したうえでの**改善候補**です。  
外部レビューで指摘されやすい **スマートコントラクト監査・オンチェーン検証** 周りを中心に、セキュリティ開示・一貫性・情報設計まで整理しています。

> 本ファイルは社内・制作用のチェックリストです。  
> **運用注意:** コンテンツルート内の `.md` は、ナビに載せなくても Mintlify の設定によっては URL で参照されることがあります。社内限定にしたい場合は、このファイルを `docs` 外（例: リポジトリ直下の `DOCUMENTATION_IMPROVEMENTS.md`）へ移すか、プロジェクトの除外ルールに従ってください。

---

## 1. 最優先：監査・オンチェーン透明性（現状の大きなギャップ）

現状、`security/security.mdx` の「Smart Contract Interaction」では設計方針（標準化・アップグレード分離など）に触れている一方、**第三者によるスマートコントラクト監査**についての記載がありません。  
DeFi / オンチェーン資金を扱うプロダクトでは、利用者・取引所・パートナーから **「誰が・いつ・何の範囲を監査したか」** がほぼ必須の期待値です。

| 改善項目 | 現状 | 推奨 |
|----------|------|------|
| 監査の有無 | 未記載 | **実施済み / 進行中 / 未実施** を明示。未実施の場合は理由と今後の計画（時期の目安が取れるなら記載） |
| 監査レポート | なし | 可能な範囲で **PDF または監査法人ページへのリンク**（公開契約に合意している場合） |
| 監査スコープ | なし | 対象コントラクト名・バージョン・コミットハッシュ・**スコープ外**（フロント、オフチェーン、運用ウォレット等）を明記 |
| 検証済みコントラクト | なし | **各チェーンの Explorer リンク**（Verified Contract）、必要なら **プロキシ実装・実装アドレス** の説明 |
| アップグレード可能性 | 「Controlled deployment and upgrades」のみ | **プロキシ／管理者権限／マルチシグ／タイムロック** の有無をユーザー向けに要約（過度な技術詳細はリンク先に） |
| 監査の限界 | リスク文書で一般的なコントラクトリスクは記載 | **「監査済みでも完全ではない」** 旨を Security または Risk に短く追記し、過信を防ぐ |

**配置案**

- `security/security.mdx` に **「Audits & On-Chain Verification」**（または同等）のセクションを新設。
- あるいは専用ページ `security/audits.mdx` を切り出し、`docs.json` の「Security & Risk」グループに追加。

## 2. セキュリティページ（`security/security.mdx`）の補強

| テーマ | 指摘・改善案 |
|--------|----------------|
| アーキ図（Mermaid） | `Operational Wallet` から `User Wallet` への矢印（`C --> A`）は、実際の **出金キュー・検証・オンチェーン送金** を省略した読み取りになり得る。**概念図であること**、または Withdrawal 経路を図に含めると誤解が減る。 |
| バグ報奨 / 脆弱性開示 | 未記載。**セキュリティ連絡先**（`security@` 等）と **想定応答時間**、HackerOne / Bugcrowd 等の有無。 |
| インシデント対応 | 未記載。**インシデント時の情報開示方針**（ステータスページ、公式チャネルのみ信頼する旨）を短く。 |
| オペレーション vs オンチェーン | 「オフチェーンでの執行・決済」が多い構成の場合、**どこまでがオンチェーン保証か** を一文で区別すると信頼説明が明確になる。 |

## 3. リスク開示（`security/risk.mdx`）の拡張候補

既存の市場・流動性・コントラ・カウンターパーティ等は一通り揃っている。以下は**追記を検討**すると外部レビューに強い。

| リスクカテゴリ | 内容例 |
|----------------|--------|
| 監査・検証の限界 | 上記1と整合させ、監査スコープ外のリスクを明示。 |
| アップグレード / 管理者権限 | ガバナンス変更・パラメータ変更がユーザー資産に与えうる影響（可能なら法務と整合）。 |
| ブリッジ・クロスチェーン | 将来マルチチェーン展開時、ブリッジ固有のリスクを追記。 |
| オラクル / 外部データ | 該当する戦略・コントラがある場合のみ。 |

## 4. プロダクト文書の精査メモ

| ファイル | 改善案 |
|----------|--------|
| `product/fund-flow.mdx` | Mermaid の `J` は「Redeem then Withdraw」に整理済み。図上は依然として簡略化のため、オンチェーン／オフチェーンの実フローと完全一致しない点に注意。 |
| `product/vaults.mdx` | 例示パラメータは `<Note>` で限定されているので良い。**実 Vault ごとの公式パラメータ一覧**へのリンク（アプリ内 or 別ページ）があると運用負荷が下がる。 |
| `overview/how-it-works.mdx` | ネットワーク対応表は**更新頻度が高い**。変更時のオーナー（誰が直すか）を決めておくとドキュメントドリフトを防げる。 |
| `faq/faq.mdx` | 監査・検証済みコントラ・バグ報告先・**ステータス / メンテ情報の確認先** の Accordion を追加推奨。 |

## 5. 法務・表現の一貫性

| ファイル | 内容 |
|----------|------|
| `legal/kyc-aml.mdx` | 「Terminate **accounts**」とあるが、他ページはウォレット主体。**「サービス利用の停止」「アクセスの終了」** 等、プロダクト実態と揃えた表現に（正本はウェブサイトの法務文書に合わせる）。 |
| Legal 各ページ | 要約＋`rondosync.com` への Card は妥当。**要約と正本の差分**が出ないよう、ウェブ更新時の同期フローを決めるとよい。 |

|------|--------|

## 8. 実装時の優先順位（提案）

1. **監査・検証済みコントラ・アドレス開示**（方針が決まっている部分から公開可能な範囲で）
2. **セキュリティ連絡先・バグ報告**
3. **図表とフローの整合**（Security Mermaid、Fund Flow の Redeem / Withdraw と実装の対応）
4. **FAQ の追記**（上記1〜2の要約）
5. **KYC 要約の表現**（アカウント表現の修正）
6. **フッター・ナビの外部リンク**（公開戦略に沿って）
